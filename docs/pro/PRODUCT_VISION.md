# AgentsOffice Pro — 产品愿景与战略规划

> **内部文档，不开源。** 本文档包含 AgentsOffice Pro 版本的核心战略定位、技术架构设计和实施路径。

---

## 一、战略定位

### 1.1 核心定位

**AgentsOffice Pro = 你的 AI 开发团队**

不是"又一个 AI 编码助手"，而是一个能自主运转的数字化专家团队。用户只需要在里程碑时看看进度报告。

### 1.2 解决的核心痛点

| 痛点 | 现有工具的问题 | AgentsOffice Pro 的方案 |
|------|--------------|----------------------|
| **决策瓶颈** | Claude Code / Codex 遇到决策就弹窗停下来，用户自己也不专业 | Agent 团队内部讨论决策，只在里程碑汇报 |
| **领域局限** | 单一模型处理所有领域，法律/设计/市场等跨领域能力不足 | 多领域专家 Agent，各司其职 |
| **模型锁定** | 绑定单一 LLM 供应商 | 多模型混用，按任务类型选最优模型 |
| **效率低下** | IDE 插件受沙箱限制，频繁弹窗确认打断工作流 | 独立后台服务，Contract 预授权，无需逐次确认 |

### 1.3 目标用户（非消费市场）

| 用户群体 | 他们的困境 | AgentsOffice Pro 的价值 |
|---------|----------|----------------------|
| 独立开发者 | 有想法但缺乏设计/法务/市场能力 | 一个人 = 一个完整团队 |
| 小型创业团队 | 请不起完整团队，预算有限 | AI 团队按需扩展，用订阅省钱 |
| 创意工作者 | 想做互动游戏/App 但不会编程 | 让 AI 团队实现创意 |
| 独立游戏开发者 | 会一点脚本但不会系统性编程 | AI 团队覆盖编程+美术+音效+测试 |

### 1.4 竞争差异化

| 竞争维度 | Cursor | Claude Code | Codex | Devin | **AgentsOffice Pro** |
|---------|--------|-------------|-------|-------|---------------------|
| 交互形态 | IDE 聊天窗口 | CLI 终端 | 异步黑箱 | Web 黑箱 | **RPG 可视化办公室** |
| Agent 数量 | 单个 | 单个 | 单个 | 单个 | **多 Agent 团队** |
| 跨领域能力 | 无 | 无 | 无 | 有限 | **多领域专家覆盖** |
| 模型灵活性 | 单模型 | 单模型 | 单模型 | 单模型 | **多模型混用** |
| 自主决策 | 弹窗确认 | 弹窗确认 | 异步 | 异步 | **Contract 预授权 + 团队自主决策** |
| 记忆进化 | 无 | 有限 | 无 | 有限 | **独立记忆仓库，持续进化** |
| 使用成本 | API 计费 | 订阅/API | 订阅 | $500/月 | **支持用户已有订阅** |

### 1.5 紫牛特质（传播核心）

> 你打开一个像素风办公室，里面坐着你的"开发团队"——项目经理、设计师、工程师、法务顾问——他们在讨论你的项目，你可以旁听，也可以插话，然后代码就自动写进了你的项目目录。

**营销策略：展示 > 描述 > 解释。** 用 GIF/视频展示像素团队讨论 + 代码同步生成。

## 二、核心架构设计

### 2.1 整体架构：CLI-as-Backend

AgentsOffice Pro 不是 IDE，不做 IDE 集成，不 fork VSCode。它是一个**独立的调度中心**，通过文件系统和 CLI 与用户的开发环境交互。

```
┌─────────────────────────────────────────────┐
│  浏览器：AgentsOffice RPG 办公室界面          │
│  （看到 Agent 团队讨论、工作、汇报）           │
└──────────────────┬──────────────────────────┘
                   │ WebSocket / REST
┌──────────────────▼──────────────────────────┐
│  AgentsOffice Backend（调度层）               │
│  ┌──────────────────────────────────────┐   │
│  │  项目经理 Agent（调度员）               │   │
│  │  - 任务分解引擎                        │   │
│  │  - Agent 协作协议                      │   │
│  │  - 检查点管理                          │   │
│  │  - 冲突解决                            │   │
│  └──────────────────────────────────────┘   │
│  ┌────────┐ ┌────────┐ ┌────────┐          │
│  │工程专家│ │设计专家│ │法务专家│ ...       │
│  │(记忆库)│ │(记忆库)│ │(记忆库)│          │
│  └────────┘ └────────┘ └────────┘          │
└──────────────────┬──────────────────────────┘
                   │ 子进程调用 (stdin/stdout)
┌──────────────────▼──────────────────────────┐
│  CLI 统一适配层                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │Claude CLI│ │Codex CLI │ │Gemini CLI│    │
│  │(订阅模式)│ │(订阅模式)│ │(订阅模式)│    │
│  └──────────┘ └──────────┘ └──────────┘    │
│  ┌──────────┐ ┌──────────┐                  │
│  │Ollama    │ │API 模式  │                  │
│  │(本地模型)│ │(fallback)│                  │
│  └──────────┘ └──────────┘                  │
└──────────────────┬──────────────────────────┘
                   │ 文件系统 / 终端 / MCP
┌──────────────────▼──────────────────────────┐
│  用户工作空间                                 │
│  ├── 项目代码目录（IDE 自动检测文件变化）      │
│  ├── Git 仓库                                │
│  └── 游戏引擎项目（Godot/Unity）              │
└─────────────────────────────────────────────┘
```

**核心原则：**
- 编辑器（VSCode/Xcode）只是"观察窗口"，AgentsOffice 才是"指挥中心"
- 用户本地装了什么编辑器都行，AgentsOffice 改文件后编辑器自动刷新
- 能力通过 MCP Server / Skills / CLI 外挂，不内置（热插拔）

### 2.2 为什么不集成 IDE

| 方案 | 代表产品 | 问题 |
|------|---------|------|
| Fork VSCode | Cursor | 维护成本极高，每次上游更新都要合并 |
| VSCode Extension | Copilot | 受沙箱限制，必须弹窗确认，无法全自动 |
| **独立后台服务** | **AgentsOffice Pro** | **不受限，Contract 预授权，真正全自动** |

VSCode Extension 受沙箱限制，每次写文件/执行命令都必须弹窗确认。这正是用户反馈"效率变低"的根因。AgentsOffice 作为独立后台服务，直接操作文件系统，不经过 IDE 权限系统。

### 2.3 CLI 订阅模式 vs API 模式

用户可以选择用已有的 CLI 订阅（省钱）或 API（灵活）：

| 模式 | 成本 | 适用场景 |
|------|------|---------|
| Claude CLI（Max 订阅） | ~$100-200/月固定 | 主力编程 Agent |
| Codex CLI（Plus/Pro 订阅） | ~$20-200/月固定 | 辅助编程 Agent |
| Gemini CLI（Google 订阅） | 免费/低价 | 简单任务 |
| Ollama 本地模型 | 免费 | 简单任务，节省配额 |
| API 调用 | 按量付费 | fallback / 特殊模型 |

**合规性设计：**
- CLI 必须在用户本地运行（用户调用自己的订阅 = 合法）
- AgentsOffice 不代理认证、不转售服务
- 同时提供 API 模式作为 fallback

**智能配额管理：**
- 简单任务 → 本地模型（不消耗配额）
- 中等任务 → 缓存 + 模板匹配（减少调用）
- 复杂任务 → CLI 调用（值得消耗配额）
- 实时监控剩余额度，接近限额自动切换

### 2.4 外部能力接入（MCP 热插拔）

AgentsOffice 是调度中心，不是能力中心。所有外部能力通过 MCP Server / Skills 接入：

| 能力 | 接入方式 | 用途 |
|------|---------|------|
| 浏览器自动化 | Playwright MCP | App 上架、短视频发布、审核处理 |
| 代码索引 | Tree-sitter | 理解项目代码结构 |
| 游戏引擎 | Godot MCP Server | 游戏开发 |
| 文档查询 | Context7 MCP | 框架/库官方文档 |
| 搜索 | WebSearch | 查找解决方案 |
| 文件操作 | FileSystem | 读写项目文件 |

**不内置任何外部工具的代码**，全部通过标准协议（MCP）对接，保持核心代码精简。

## 三、两种工作模式

### 3.1 模式 A：用户参与模式

用户全程在场，随时介入、指导、修改。类似当前开源版的交互方式。

### 3.2 模式 B：全自动托管模式

Agent 团队完全自主运行，用户不需要实时参与。

#### 3.2.1 为什么能做到真正的"全自动"

| | VSCode 里的 Claude | AgentsOffice 托管模式 |
|---|---|---|
| 架构 | IDE 插件，受沙箱限制 | 独立后台服务，无沙箱 |
| 写文件 | 弹窗确认 | Contract 预授权，直接写 |
| 执行命令 | 弹窗确认 | 白名单内直接执行 |
| 遇到歧义 | 弹窗问用户 | Agent 团队内部讨论决策 |
| 用户体验 | 不断被打断 | **真正的"甩手掌柜"** |

#### 3.2.2 状态机设计

```
CONTRACTING ──→ PLANNING ──→ PLAN_REVIEW ──→ EXECUTING ⇄ CHECKPOINT
                                ↑                 ↓
                                └── REVIEW ←── BLOCKED
                                      ↓
                                  COMPLETED
```

| 状态 | 说明 |
|------|------|
| CONTRACTING | 和用户签约：明确目标、约束、权限 |
| PLANNING | Agent 团队制定执行计划 |
| PLAN_REVIEW | 检查点：用户审核计划后批准 |
| EXECUTING | Agent 团队自主执行 |
| CHECKPOINT | 里程碑检查点，向用户汇报 |
| BLOCKED | 遇到超出授权范围的问题，等待用户决策 |
| REVIEW | 里程碑审查 |
| COMPLETED | 项目完成 |

#### 3.2.3 Contract（甲乙方协议书）

托管模式的成败，90% 取决于 Contract 的质量。

```yaml
AgentOffice_Project_Contract:

  # 项目目标
  project_goal:
    description: "一句话描述最终产品"
    success_criteria:
      - "可验证的验收标准 1"
      - "可验证的验收标准 2"
    non_goals:
      - "明确不做的事情"  # 比"做什么"更重要

  # 约束条件
  constraints:
    technology:
      language: "Python 3.11+"
      framework: "FastAPI"
      must_use: ["SQLAlchemy", "Pydantic"]
      must_not_use: ["Django", "Flask"]
    scope:
      files_allowed: ["app/", "tests/"]
      files_forbidden: ["config/production.py", ".env"]
    quality:
      test_coverage_minimum: 80
      must_pass: ["mypy", "ruff", "pytest"]

  # 里程碑
  milestones:
    - name: "数据模型完成"
      deliverables: ["models.py", "migration 脚本", "模型测试"]
      checkpoint_type: "L2"
    - name: "API 端点完成"
      deliverables: ["路由文件", "API 测试", "接口文档"]
      checkpoint_type: "L2"

  # 决策授权
  agent_autonomy:
    can_do_without_asking:
      - "在允许目录下读写文件"
      - "执行 git add / commit"
      - "运行测试和 lint"
      - "安装 contract 中批准的依赖"
    must_ask:
      - "新增未批准的依赖"
      - "修改数据库 schema"
      - "改变 API 接口契约"
    cannot_do:
      - "修改 contract 本身"
      - "访问 scope 之外的文件"
      - "执行 rm -rf 等危险命令"
      - "执行不可逆的生产操作"

  # 沟通协议
  communication:
    progress_update: "每个里程碑"
    blocking_notification: "WebSocket + 邮件"
    final_delivery: "完整代码 + 测试报告 + 文档 + 变更日志"
```

#### 3.2.4 四层检查点

| 层级 | 触发时机 | 处理方式 | 打扰用户？ |
|------|---------|---------|-----------|
| **L0 自动验证** | 每次代码提交 | 测试/lint，失败自动修复（最多重试3次） | 否 |
| **L1 团队审查** | 每个任务完成 | Agent 间互审，记录决策理由 | 否 |
| **L2 里程碑** | 功能模块完成 | 进度报告 + 演示 + 用户确认 | **是** |
| **L3 紧急中断** | 歧义/安全/超预算 | 立即通知，等待用户决策 | **是，立即** |

80% 的情况下，只有 L0 和 L1 在运行，用户完全不被打扰。

#### 3.2.5 检查点报告模板

```
核心信息（5秒看完）：
  状态：🟢 正常推进
  进度：3/7 里程碑完成（43%）
  摘要：完成了用户认证模块，正在做数据 API

详细信息（想看再看）：
  ✅ 用户注册/登录 API（含测试）
  ✅ JWT 认证中间件
  🔄 数据 CRUD API（预计 2 小时完成）
  ⏳ 前端页面
  ⏳ 部署配置

需要用户决策（如有）：
  "数据库用 PostgreSQL 还是 SQLite？"
  Agent 建议：SQLite，因为 contract 定义为原型项目
```

#### 3.2.6 信任阶梯

不一步到位给全自动，渐进解锁：

```
Level 1 → 用户全程在场观察 Agent 工作方式
Level 2 → 简单任务自动执行，复杂任务问用户
Level 3 → 1-2小时短周期托管，完成后审查
Level 4 → 完整功能模块托管，每天汇报
Level 5 → 项目级完全托管
```

用户在每层成功完成一定数量的任务后，系统自动解锁下一层。

#### 3.2.7 防止 Agent 跑偏（Via Negativa 原则）

"不做什么"比"做什么"更重要：

- 不得修改 Contract 中未提及的文件
- 不得引入 Contract 中未批准的依赖
- 不得删除现有通过的测试
- 不得改变已确认的接口契约
- 执行 Agent 和审查 Agent 必须是不同的 Agent
- 可逆操作 → 自动执行；不可逆操作 → 必须进入 BLOCKED 等待用户

## 四、应用场景扩展

### 4.1 核心场景（优先）

**独立开发者 / 小团队的全栈项目开发**

### 4.2 扩展场景（飞轮转起来后逐步开放）

| 场景 | Agent 团队配置 | 外部能力 |
|------|--------------|---------|
| 独立游戏开发 | 策划 + 程序 + 美术 + 音效 + 测试 | Godot MCP |
| App 开发 + 上架 | 产品 + 开发 + 测试 + 运营 | Playwright MCP（应用商店操作） |
| 短视频创作 | 编剧 + 剪辑 + 运营 | Playwright MCP（平台发布） |
| 小说创作 | 作者 + 编辑 + 校对 | 文件系统 |
| 技术文档 | 技术写作 + 审校 | Context7 MCP |

### 4.3 游戏引擎集成路径

| 优先级 | 引擎 | 原因 |
|--------|------|------|
| 1 | **Godot** | 开源、GDScript 简单、AI 生成质量高、支持命令行 |
| 2 | Unity | 生态大，C# 较复杂 |
| 3 | Unreal | C++ 门槛最高，Blueprint 需特殊处理 |

## 五、飞轮效应与护城河

### 5.1 飞轮

```
更多用户使用
    ↓
更多 Agent 模板和技能插件（社区贡献）
    ↓
新用户更快上手、解决更多场景
    ↓
Agent 记忆积累，团队越来越"聪明"
    ↓
用户依赖度增加，切换成本提高
    ↓
口碑传播
    ↓
更多用户使用（回到起点，但更强）
```

### 5.2 护城河（真正的壁垒）

| 护城河 | 说明 |
|--------|------|
| **项目知识积累** | Agent 记忆仓库随使用越来越丰富，迁移成本高 |
| **协作协议** | 多 Agent 协调的调度质量和冲突解决能力 |
| **技能生态** | MCP Server / Skills 插件生态的网络效应 |
| **品牌认知** | "可视化 AI 团队协作" = AgentsOffice |

CLI 包装层没有壁垒（谁都能写），**护城河在调度层和知识层**。

### 5.3 反脆弱设计

| 风险 | 应对策略 |
|------|---------|
| LLM 供应商提价/改接口 | 多供应商冗余，任何一个挂掉系统降级但不崩溃 |
| 供应商推出竞品 | 核心价值在协作协议和项目记忆，不在 CLI 调用 |
| AI 给出错误建议 | 高风险领域（法律/医疗）强制免责声明 + 人工确认 |
| 复杂性爆炸 | 四层检查点 + Via Negativa 约束 |

**杠铃策略**：90% 资源放已验证的编程协作场景，10% 探索创意场景。

## 六、实施路径

| 阶段 | 时间 | 核心交付 | 目标 |
|------|------|---------|------|
| **Phase 1：核心飞轮** | 0-6月 | CLI 适配层 + 基础状态机 + Contract MVP + 用户参与模式 | 飞轮开始转，10个深度用户验证 |
| **Phase 2：能力扩展** | 6-12月 | 托管模式 + 四层检查点 + 记忆系统 + Agent 间审查 | 飞轮加速，100个活跃用户 |
| **Phase 3：场景拓展** | 12-18月 | 游戏引擎集成（Godot）+ 第二应用场景 + Pro 商业化 | 证明蓝海，可持续商业模式 |

### 需要验证的关键假设

| 假设 | 验证方法 | 优先级 |
|------|---------|--------|
| 用户信任 AI 团队做跨领域决策 | 原型测试 + 用户访谈 | P0 |
| 多 Agent 协调可靠性达到生产级 | 技术原型 + 压力测试 | P0 |
| 可视化协作降低使用门槛 | A/B 测试 | P1 |
| Agent 记忆进化产生可感知质量提升 | 纵向跟踪 | P1 |
| 非编程场景有足够需求 | 市场调研 | P2 |

## 七、刺猬原则

> **让复杂的多 Agent AI 协作变得可见、可控、有趣。**

- 最擅长什么 → 多 Agent 可视化协作
- 经济引擎是什么 → 用户为"节省的时间 × 项目复杂度"付费
- 热爱什么 → 让枯燥的技术交互变得有趣和直观
